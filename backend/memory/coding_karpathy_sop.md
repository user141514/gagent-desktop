# Karpathy 编码准则

> 来源: F:\andrej-karpathy-skills | 用于减少LLM编码常见错误

## 1. 动手前思考 (Think Before Coding)

**不假设，不隐藏困惑，显式陈述权衡。**

执行前必须：
- 显式陈述假设。不确定就问。
- 存在多种理解时，全部呈现，不要静默选择。
- 有更简单方案时说出来。必要时反驳。
- 有不清楚的地方，停下来，说出困惑，询问。

## 2. 简洁优先 (Simplicity First)

**最小代码解决问题。不写投机性代码。**

- 不写超需求的功能。
- 单次使用的代码不做抽象。
- 没人要求的"灵活性"或"可配置性"不写。
- 不可能发生的场景不写错误处理。
- 如果200行能写成50行，重写。

自问："资深工程师会觉得这太复杂吗？" 如果是，简化。

## 3. 手术式修改 (Surgical Changes)

**只改必要的。只清理自己制造的混乱。**

编辑现有代码时：
- 不"改进"相邻代码、注释或格式。
- 不重构没坏的东西。
- 匹配现有风格，即使你不会那样写。
- 发现无关死代码，提一下，不要删除。

当你的修改产生孤儿代码时：
- 删除你的修改导致无用的 imports/variables/functions。
- 不要删除预先存在的死代码，除非被要求。

检验：每行修改都应能追溯到用户请求。

## 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环直到验证。**

将任务转化为可验证目标：
- "添加验证" → "为无效输入写测试，然后让它们通过"
- "修复bug" → "写一个复现它的测试，然后让它通过"
- "重构X" → "确保重构前后测试都通过"

多步骤任务，陈述简要计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强成功标准让你能独立循环。弱标准（"让它工作"）需要不断澄清。

## 5. 验证铁律 (Verification Rules)

> 整合自 verify_sop.md | 编码后必须验证

**核心原则：** 无工具输出的PASS = SKIP。读代码不是验证。

### 编码后必须执行

|| 产物类型 | 必做验证 |
|---|---|
| 脚本/CLI | 执行 → 检查stdout/stderr/exit code → 边界输入 |
| API/服务 | 调用endpoint → 响应形状 → 错误输入 |
| 配置/文档 | file_read完整内容 → 格式语法 |
| Bug修复 | 复现原bug → 验证修复 → 回归测试 |

### 对抗性探测（至少运行一个）

- 边界值(0/空/超长/unicode)
- 幂等性(同一操作两次)
- 缺失依赖

### 自检清单

```
□ 每步有命令输出？
□ 跑了对抗探测？
□ 独立验证了(不是只看测试套件)？
```

---

**准则生效指标：** diff中不必要变更减少，过度复杂导致的重写减少，澄清问题在实现前而非错误后出现，验证工具输出可追溯。
