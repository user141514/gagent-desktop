# 理解能力增强原则

> 来源: F:\claude-code-best-practice (Boris Cherny, Thariq 等实践)
> 用于增强模型理解质量，减少认知盲区

## 1. 多视角验证 (Multiple Context Windows)

**单一上下文存在认知盲区，多角度验证更可靠。**

执行复杂任务时：
- 自己写的代码，自己很难发现自己引入的bug
- 多个独立上下文窗口比单一上下文更有效发现问题
- 类似工程团队：作者造成bug，审查者更容易发现

实践：
- 重要修改后，用"审查者视角"重新审视
- 长任务完成后，主动问"有没有遗漏"
- 复杂逻辑考虑分两个独立检查轮

---

## 2. 渐进式披露 (Progressive Disclosure)

**信息不是越多越好，按需加载减少认知负担。**

理解复杂系统时：
- 先获取高层结构，再按需深入细节
- 告知模型有什么文件可读，而非一次全读完
- 文件系统本身是上下文工程工具

实践：
- 先 `ls` 再针对性 `file_read`
- 长文档优先读目录/开头，定位后再深入
- 避免一次性读取超过必要的信息量

---

## 3. 目标驱动而非步骤驱动 (Goal-Driven, Not Railroading)

**给目标和约束，让模型决定路径。**

理解任务时：
- 说明"要达成什么"和"不能做什么"
- 不要规定具体步骤，给模型灵活性适应情况
- 过度具体的指令会限制模型的判断力

实践：
- ✗ "先读A文件，再搜索B关键词，然后..."
- ✓ "找出导致X的原因，注意不要修改生产配置"
- ✗ "检查第50-100行有没有问题"
- ✓ "检查这段代码的边界情况处理"

---

## 4. Gotchas积累 (Pitfall Collection)

**从失败中提取的教训最高信号。**

理解新领域时：
- 主动记录"踩过的坑"
- Gotchas比正面知识更有价值
- 每次失败后问"这个坑怎么避免"

实践：
- 遇到错误时，记录根本原因而非仅修复
- 常见坑模式：环境差异、版本兼容、隐藏依赖
- Gotchas应该持续更新，不是写完就忘

---

## 5. 验证闭环 (Verification Loop)

**理解必须有验证方式，否则不可信。**

完成任务后：
- 后端工作：运行服务端到端测试
- 前端工作：浏览器交互验证
- 脚本：实际执行看结果

验证形式因任务而异：
- 有测试套件 → 跑测试
- 无测试 → 手动验证关键路径
- 可视化输出 → 截图/录屏确认

---

## 6. 快速恢复上下文 (Quick Context Recovery)

**长任务中断后需要快速重建理解。**

实践：
- 长任务期间主动生成简短摘要
- 返回时说明"之前做了什么，接下来做什么"
- 摘要格式：`* 完成了... * 下一步：...`

---

## 与现有准则的关系

| 本SOP原则 | 对应karpathy准则 | 关系 |
|-----------|------------------|------|
| 多视角验证 | 目标驱动执行 | 补充验证方法 |
| 渐进式披露 | 简洁优先 | 一致，信息层面 |
| 目标驱动 | 动手前思考 | 一致，指令层面 |
| Gotchas积累 | 手术式修改 | 补充知识积累 |
| 验证闭环 | 目标驱动执行 | 一致，执行层面 |

---

**原则生效指标：** 返工减少、首次理解准确率提升、长任务恢复时间缩短。
